## 🎯 このプロジェクトの目的

- **BL/TL系イラストレーター候補**をXのフォローグラフから自動発掘する
- 候補探索（スクレイピング）→ 一次判定（スカウト）→ 画力確定（オーナー）→ 連絡（連絡担当）の各フェーズを最適なツールで分担
- 既存Notion DB（複数プロジェクトで使用中）をリファクタリングしてデータの純度を高める

## 📐 システム全体構成

### アーキテクチャ

```
[X（Twitter）フォローグラフ]
         ↓ スクレイパー（順次一括・オンデマンド）
[Supabase（真のデータストア・直接編集なし）]
         ↓ 10分おき同期      ↑ 10分おき同期
[Google Sheets（スカウト作業場所）]   [Notion（オーナー・連絡担当の作業場所）]
                                           ↓ relation
                                    [Notion 案件管理DB（別DB）]
```

### データストアの役割分担

| ストア | 役割 | 書き込み主体 |
|---|---|---|
| **Supabase** | 真のデータストア、集計・検索・分析のバックエンド | スクレイパー・同期ジョブのみ（直接編集禁止） |
| **Google Sheets** | スカウトの作業場所 | スカウト |
| **Notion** | オーナー・連絡担当の作業場所、案件管理DBとの連携 | オーナー・連絡担当 |

### 同期パターン：**パターンB**（フィールド別主体）

- **Supabase主導（基本情報）**：作家名・Xリンク・フォロワー数・検出元・first_detected_at・last_seen_at
- **Notion主導（運用情報）**：マスターステータス・ランク・オーナー確認・絵柄タグ・備考・連絡した日・連絡した人・メール・ポートフォリオサイト・その他連絡先・クレジット名義・ジャンル
- **Pixivリンク**：初回のみスクレイパーが書き込み、以降はNotion主導（オーナー編集が優先）
- 同じフィールドを両側で編集することはない

### 同期頻度：**10分おき**

- リアルタイム不要
- Notion ↔ Supabase はポーリング方式（Notion APIのwebhookが弱いため）
- `last_edited_time > 前回同期時刻` で変更検知

### 同期ジョブの実行環境：**GitHub Actions**

- **同一リポジトリ内（`illustrator-scraper`）の `.github/workflows/` で管理**
- スクレイパーと同期ジョブを一元管理
- cron で10分おき実行
- シークレットはGitHub Secretsで管理

### 失敗通知：**1時間おきにまとめてメール送信**

- 宛先：加藤さん（`ai@roadie.co.jp` または `takuyakato.work@gmail.com`）
- 失敗0件ならスキップ
- 失敗が発生したらSupabaseの `sync_failures` テーブルに蓄積
- 自動リトライ（最大10回）、それでも失敗するものを通知

## 🗄 Supabase運用ポリシー

### 5つの原則

1. **書き込みは3経路のみ**
   - スクレイパー
   - Google Sheets同期ジョブ
   - Notion同期ジョブ

2. **Supabase Studioでの直接編集禁止**（通常運用）

3. **管理者例外**（加藤さんのみ）
   - マイグレーション
   - 同期失敗の手動修復
   - スキーマ変更

4. **読み取りは自由**

5. **変更ログは全て自動記録**（`updated_at`、`scraping_logs`、`sync_failures`）

## 🗃 Supabase テーブル設計（illustrators）

### 1テーブル + is_illustrator フラグ方式

非イラストレーターもSupabaseに記録（二度取り防止）、ただしNotionには同期しない。

### is_illustrator の3状態

| 状態 | 意味 | スプレッドシート | Notion |
|---|---|---|---|
| `null` | 未判定（スカウト対応待ち） | 表示 | 非表示 |
| `true` | イラストレーター確定 | 非表示 | 表示 |
| `false` | 非イラストレーター or 除外 | 非表示 | 非表示 |

### アクティブカラム（17項目・Notion同期対象）

1. 作家名（title）
2. ID（unique_id・自動）
3. マスターステータス（status、7値・新設）
4. **ランク**（select、S/A/B/C）  ※ マイグレーション時に既存ランク値をそのままコピー、オーナーが必要に応じて変更
5. **オーナー確認**（multi_select、北條/三村/加藤）  ※ マイグレーション後は全員空（要確認）
6. 絵柄タグ（multi_select、イケメン/リアル/デフォルメ/クセ強）
7. ジャンル（multi_select、BLサンド/Capuri/Berryfeel/Webtoon/アシスタント）
8. Xリンク（url）
9. Pixivリンク（url）
10. ポートフォリオサイト（url、新設）
11. その他連絡先（url）
12. メール（email、旧「メアド」と統合）
13. クレジット名義（rich_text、旧「クレジット希望」を改修）
14. 連絡した日（date）  ※ 最新連絡日で上書き更新
15. **連絡した人**（multi_select） ※ 拡張可能な選択肢、複数人記録可
16. 備考（rich_text）
17. 最終更新日時（自動）

### オーナーが変更前のランクを見たい場合

- **`migration_snapshot`（JSONB、非表示）** に全オリジナルデータが保存されている
- 必要な場合は Supabase Studio でクエリ参照
- 日常運用では変更履歴を別カラムに残さない（シンプル運用優先）

### 非表示カラム（Supabase個別保持、Notion非同期）

旧プロパティは**Supabaseに個別カラムで残す**（JSONB統合はしない）。
必要になったら後から「このカラムもNotionに反映したい」と実装できる設計。

- `legacy_status`（旧ステータス）
- `legacy_status_1`（旧ステータス 1）
- `legacy_contact_status`（旧連絡状況）
- `legacy_capuri_berryfeel_search`（旧Capuri/BerryFeel探し）
- `legacy_mimura_comment`、`legacy_hojo_comment`（三村コメント・北條コメント）
- `legacy_mimura_points`、`legacy_hojo_points`（三村点数・北條点数）
- `legacy_found_date`、`legacy_found_by`（見つけた日・見つけた人）
- `legacy_start_date`、`legacy_end_date`（開始日・終了予定日）
- `legacy_capuri_request_id`（Capuri依頼のNotion page ID）
- `legacy_mail_alt`（メアド）
- `legacy_recontact_time`（再連絡時期、rich_text）
- `legacy_rejection_reason`（断られた理由、multi_select→配列）
- **migration_snapshot**（JSONB、念のためマイグレーション時の完全スナップショット）

### 完全削除ポリシー

**❌ 完全削除はなし**。全プロパティのデータを保持する。不要データはSupabase側に保持しつつ、Notionには同期しない。

## 📋 マスターステータス（7値）

### 値とグループ

```
📋 候補段階
  1. 候補

📞 進行中
  2. 連絡中

🟡 定期見直し対象
  3. 返信なし        ← 連絡中から30日経過で自動遷移
  4. 多忙辞退        ← 時期の問題
  5. 条件次第        ← 金額・条件の問題

✅ 完了
  6. 依頼成功        ← シード抽出対象

🚫 終了
  7. 依頼不可
```

### 自動遷移ルール

- **連絡中 → 返信なし**（連絡した日から30日経過で自動）：GitHub Actionsで定期実行
- 他は全て手動（担当者判断）

### ループOK

- 返信なし → 連絡中 → また返信なし → 連絡中 … のループは許容
- 再連絡時に「連絡した日」を**上書き更新**（最新日に書き換え）

### 備考の書き方（運用ルール）

```
2026-04-22 [多忙辞退] 連載明けの秋頃なら可能
2026-05-15 [条件次第] 1ページ10万以上希望、短納期NG
2026-06-01 [再連絡予定] 7月以降で再アプローチ
```

- 日付 + [ステータス変更時のラベル] + 内容
- 追記式（過去の内容は残す）

## 🏅 ランク（S/A/B/C）とプロジェクト対応

### ランクの意味

| ランク | 画力 | 主な依頼先 |
|---|---|---|
| S | 最高 | BerryFeelのみ |
| A | 高 | CAPURI中心（BerryFeelも可） |
| B | 標準 | BLサンド |
| C | 依頼しない | ー（イラストレーターだが対象外） |

### マイグレーション時のランク処理

**全レコードに対して**:
- `rank`（新） ← 旧ランクをそのままコピー
- `owner_confirmed_by`（オーナー確認） ← 空配列（要確認扱い）

**例外：BerryFeel別DBから統合したレコード**:
- `rank` ← `'S'`（BerryFeel採用済み＝最高ランク相当、既存ランクがあっても **S で上書き**）
- `owner_confirmed_by` ← `['北條']`（北條さんが確認済み扱いとする、再確認不要）
- `genres` に `Berryfeel` を必ず付与（既存レコードに統合する場合は配列にマージ）

### オーナー確認フラグの運用

- **空** = まだオーナーが見直していない
- **北條/三村/加藤** = 見直して確認済み

オーナーは「オーナー確認=空」の人を順次見直し、以下のアクションを取る：
- ランクが正しい → オーナー確認に自分の名前を追加
- ランク変更したい → ランクを変更 + オーナー確認に自分の名前を追加

※ 変更前のランク値を参照する必要がある場合は `migration_snapshot` JSONB を使用。

## 👥 連絡した人（multi_select）

### 設計

- **multi_select**（複数人記録可能）
- 選択肢は**拡張可能**（新しい担当者が追加されたらオプション追加）
- 既存選択肢の例：李・吉澤・長野・木村・野末・赤坂・北條・荻野・加藤・本人からの応募・及川・伊藤 など
- Supabaseでは `contacted_by TEXT[]` として保存

### オーナー確認との違い

- **オーナー確認**（`owner_confirmed_by`）：画力判定したオーナー3名（北條/三村/加藤）の記録
- **連絡した人**（`contacted_by`）：実際に連絡した担当者（オーナー以外のスタッフも含む、拡張可能）

## 🔍 スクレイパー判定ロジック

### 実行モード

**週1バッチではなく順次一括（オンデマンド実行）**

- **初回フルスキャン**：全シードのフォロー欄を順次取得（数日〜1週間かけて）
- **差分スキャン**：新規シード追加時・月1の再確認
- **手動実行**：加藤さんが必要時に実行

### シード抽出条件

```sql
SELECT * FROM illustrators
WHERE rank IN ('S', 'A')
  AND master_status = '依頼成功'
  AND ('BLサンド' = ANY(genres)
    OR 'Capuri' = ANY(genres)
    OR 'Berryfeel' = ANY(genres))
```

マイグレーション時に依頼成功作家の新ランクに旧ランクがコピーされているので、マイグレーション直後から動く。

### Phase 1（運用開始時）：シンプルルール

```
1. AI絵師キーワード含む → 除外（is_illustrator=false）
2. Pixivリンクあり → 通過（is_illustrator=null、スカウト判定待ち）
3. Pixivリンクなし → 除外（is_illustrator=false、二度取り防止）
```

### 重複チェック（重要）

```
候補1件取得：
  1. Supabase.illustrators テーブルで x_username を検索
  2. 既存あり（既に登録済み）:
     → last_seen_at（最終検出日）を更新するだけ
     → 候補プール（Google Sheets）には出さない
     → スカウトは「既存イラストレーター」を見ない
  3. 既存なし（新規）:
     → is_illustrator = null で Supabase に INSERT
     → Google Sheets に表示（スカウト判定待ち）
```

### Xリンクの正規化（必須）

マイグレーションとスクレイパーで共通使用する正規化関数：

| 入力 | 正規化後（x_username） |
|---|---|
| `https://x.com/example` | `example` |
| `https://twitter.com/example` | `example` |
| `https://x.com/example/` | `example` |
| `https://x.com/example?ref=xxx` | `example` |
| `@example` | `example` |
| `Example`（大文字混じり） | `example` |

正規化後の `x_username` を一意キーに使う。

### ポートフォリオリンクの自動振り分け

既存「その他連絡先」のURLを、マイグレーション時に以下のパターンでポートフォリオサイトに自動振り分け：

| URLパターン | 振り分け先 |
|---|---|
| `lit.link/*` | portfolio_link |
| `potofu.me/*`、`potofu.jp/*` | portfolio_link |
| `skeb.jp/*` | portfolio_link |
| `booth.pm/*` | portfolio_link |
| それ以外 | other_contact（元のまま） |

### Phase 2〜4 ロードマップ

- Phase 2（運用後3ヶ月）：データ蓄積（500〜1,000件の判定データ）
- Phase 3（3〜6ヶ月目）：Claude APIでパターン抽出、ルール拡張
- Phase 4（6ヶ月以降）：4段階スコアリング方式導入、LLMで境界線ケース自動判定

### スプレッドシート列

| 列 | ヘッダー | 入力者 | 説明 |
|---|---|---|---|
| A | 検出日 | 自動 | バッチ実行日 |
| B | 検出元 | 自動 | シードXアカウント |
| C | Xアカウント | 自動 | |
| D | 表示名 | 自動 | |
| E | プロフィール | 自動 | bio（最大500文字） |
| F | フォロワー数 | 自動 | |
| G | Pixivリンク | 自動 | |
| H | ポートフォリオ | 自動 | |

| **J** | **判定** | **スカウト** | 空 / 「イラストレーターじゃない」 |
| **K** | **仮ランク** | **スカウト** | S/A/B/C |
| L | コメント | スカウト | 自由記述 |
| M | 同期状態 | 自動 | 未同期/同期済み/同期失敗 |

### スカウトの作業ルール

- **イラストレーターだ**：仮ランク（S/A/B/C）を入力
- **イラストレーターじゃない**：J列で「イラストレーターじゃない」を選択
- **判断保留**：空欄のまま置く（次回判定）

### ソート順

- **検出日順**（降順、新しい順）
- フォロワー数は気にしない

### 判定の確定

自動同期で即時反映（確定フラグは不要）。スカウトが入力すると10分以内にSupabase→Notionへ反映される。

### スカウトは複数人想定

- 誰が判定したかの記録は**不要**

### 判定ガイドライン

運用マニュアル（別途作成）で明文化。個別判定なので、スカウトの裁量に委ねる部分が大きい。

## 🖥 Notion ビュー設計（3つ）

### ビュー1：🔍 オーナー確認用

- **フィルタ**：`オーナー確認 = 空`
- **グループ化**：マスターステータスで分類（優先度順表示）
- **ソート**：グループ内はランク降順（S → A → B → C）
- **用途**：オーナー（北條・三村・加藤）が順次ランクを確認・調整
- **列**：作家名 / マスターステータス / ランク / オーナー確認 / 絵柄タグ / ジャンル / 連絡した日 / Xリンク / Pixivリンク / ポートフォリオサイト / 備考

### ビュー2：📞 連絡担当用

- **フィルタ**：`マスターステータス IN (候補, 連絡中, 返信なし) AND オーナー確認 ≠ 空 AND ランク IN (S, A, B)`
- **用途**：連絡対象選定、進行中管理
- **列**：作家名 / マスターステータス / ランク / ジャンル / 絵柄タグ / 連絡した日 / 連絡した人 / Xリンク / メール / その他連絡先 / クレジット名義 / 備考

### ビュー3：📋 全件表示（検索用）

- **フィルタ**：なし
- **用途**：俯瞰、突発的な検索
- **列**：作家名 / マスターステータス / ランク / 絵柄タグ / ジャンル / Xリンク / メール / 最終更新日時

## 🚚 マイグレーション方針

### データ保持

**❌ 完全削除なし**。全プロパティのデータをSupabaseに保持する。

### 流用

| 既存 | → | 新 |
|---|---|---|
| 連絡状況 | → | マスターステータス（値変換+リネーム） |
| クレジット希望 | → | クレジット名義（rich_text化） |
| ランク | → | ランク（既存値をそのままコピー、オーナーが以降変更可） |
| 連絡した人（select） | → | 連絡した人（multi_select化、既存値は配列の1要素として保持） |

### マスターステータスへの変換マップ

優先順位：`連絡状況` > `ステータス 1` > `ステータス`

詳細マップは `docs/architecture/03_マイグレーション手順.md` で定義。

### Berryfeel別DB統合

- レコードをメインDBに吸収
- ジャンル=Berryfeelタグで識別
- **ランク=`S`** で統合（BerryFeel採用済み＝最高ランク相当、既存ランクがあっても S で上書き）
- **オーナー確認=`['北條']`** で統合（北條さんが既に確認済み扱いとする、再確認不要）
- 別DBは `[ARCHIVED 2026-04-XX]` でアーカイブ（削除はしない）

### マイグレーション後の初期状態

| プロパティ | 初期値 |
|---|---|
| `rank` | 旧ランクをそのままコピー（全員） |
| `owner_confirmed_by` | 空配列（全員要確認扱い）※ただしBerryFeel別DB統合レコードは `['北條']` |
| `is_illustrator` | true（全員イラストレーター扱い） |
| `master_status` | 変換マップで設定 |
| `style_tags` | 空（オーナーが今後入力） |
| `contacted_by` | 既存「連絡した人」の値を配列化（例: `["李"]`） |
| `migration_snapshot` | 元のNotionページ全体をJSONBで保存 |

### マイグレーション後のオーナー初期作業

ビュー1「オーナー確認用」には全レコードが並ぶ。優先順位順で順次処理：

1. 🔥 マスターステータス=依頼成功、ランクS（最優先、シード候補）
2. ⭐ マスターステータス=依頼成功、ランクA
3. マスターステータス=依頼成功、ランクB/C
4. マスターステータス=候補/多忙辞退/条件次第、ランクあり
5. その他（依頼不可等）
