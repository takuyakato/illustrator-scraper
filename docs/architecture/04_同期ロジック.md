このドキュメントは、Supabase を真のデータストアとして、Notion / Google Sheets / スクレイパーを周辺UI・入力経路として扱うための**同期ロジックの技術仕様書**です。
実装時にそのまま参照できる詳細度で書いています。

---

## 1. 概要

### 1.1 同期の目的

- **Supabase を真のデータストア** として、全データを一元管理する
- **Notion は「オーナー・連絡担当の作業場所」** として、人が触れる UI を提供する
- **Google Sheets は「スカウトの作業場所」** として、候補プールの一次判定 UI を提供する
- **スクレイパーは Supabase に直接書き込む** 入力経路として機能する

Supabase Studio での直接編集は **通常運用では禁止**（管理者例外：加藤さん）。書き込み経路は以下の3つのみ。

```
[スクレイパー] ──直接INSERT/UPSERT──┐
                                    ↓
[Google Sheets]──同期ジョブ(S↔Sh)──→ [Supabase] ←──同期ジョブ(S↔N)── [Notion]
                                    ↑
                          [自動遷移ジョブ(30日→返信なし)]
```

### 1.2 設計原則：パターンB（フィールド別主体）

同じフィールドを両側で編集しない設計にして、**どちらを正とするか** の競合を原理的に発生させない。

| 区分 | フィールド | 主体 | 書き込み元 |
|---|---|---|---|
| **Supabase主導**<br>（基本情報・機械的に入る） | `artist_name`<br>`x_link`<br>`x_username`<br>`follower_count`<br>`detected_from`<br>`first_detected_at`<br>`last_seen_at`<br>`bio` / `display_name` | Supabase | スクレイパー |
| **Notion主導**<br>（運用情報・人が編集する） | `master_status`<br>`rank`<br>`owner_confirmed_by`<br>`style_tags`<br>`genres`<br>`note`<br>`contacted_at`<br>`contacted_by`<br>`email`<br>`portfolio_link`<br>`other_contact`<br>`credit_name`<br>`pixiv_link`※ | Notion | 人（オーナー・連絡担当） |
| **Sheets主導**<br>（候補プールの一次判定） | `is_illustrator`（`true`/`false`）<br>`rank`（仮ランク S/A/B/C）<br>`scout_comment` | Sheets | 人（スカウト） |

※ `pixiv_link` は初回のみスクレイパーが書き込み、以降は Notion 主導（オーナー編集が優先）。

**競合の考え方**:
- 基本的に同じフィールドを両側で編集しないので、競合は発生しない
- 万一発生したら「Notion の `last_edited_time` が新しい方を勝たせる」をデフォルトとする
- スカウトが入力した `rank` は Sheets→Supabase→Notion へ流れるが、Notion 側でオーナーが上書きした時点で「オーナーが正」になる

### 1.3 同期頻度：10分おき

- **リアルタイム性は不要**（スカウトも連絡担当も即時反映を求めない）
- **Notion API の webhook が弱い** ため、ポーリング方式を採用
- **GitHub Actions の cron** で10分おきに実行

```
cron: '*/10 * * * *'  # 毎時 0/10/20/30/40/50 分
```

### 1.4 失敗通知：Notion ログページ（メールではなく）

合意事項リスト v2.2 で「メール通知は廃止、Notion ログページに統一」と決定済み。

- **1時間おきにまとめて通知**（失敗0件ならスキップ）
- 通知先は Notion の1ページ（フラット構造）
- ページIDは環境変数 `NOTION_SYNC_LOG_PAGE_ID` で指定

### 1.5 同期ジョブ一覧

| # | ジョブ名 | 方向 | 頻度 | 実行場所 |
|---|---|---|---|---|
| 1 | スクレイパー → Supabase | 一方向 | オンデマンド | スクレイパー内部（同期ジョブではない） |
| 2 | Supabase → Notion | Push | 10分おき | GitHub Actions |
| 3 | Notion → Supabase | Pull（ポーリング） | 10分おき | GitHub Actions |
| 4 | Supabase → Google Sheets | Push | 10分おき | GitHub Actions |
| 5 | Google Sheets → Supabase | Pull（ポーリング） | 10分おき | GitHub Actions |
| 6 | 自動遷移（30日で返信なし） | Supabase内部 | 1日1回 | GitHub Actions |
| 7 | 失敗通知 | Supabase → Notion | 1時間おき | GitHub Actions |

## 2. 5つの同期ジョブ

### 2-1. スクレイパー → Supabase（書き込み直接）

#### 2-1-1. 概要

これは **同期ジョブではなくスクレイパー実装の一部**。Phase 3 のスクレイパーが X のフォローグラフから候補を取得した際、Supabase API で直接 UPSERT する。

- Supabase の `illustrators` テーブルに `ON CONFLICT (x_username) DO UPDATE` で書き込む
- 既存レコードは `last_seen_at` と `detected_from` のみ更新（基本情報は上書きしない）
- 新規レコードは `is_illustrator = NULL` で INSERT（スカウト判定待ち）

詳細は `docs/architecture/05_スクレイパー仕様.md`（Phase 3 で作成予定）で定義するが、同期ロジックの観点から押さえるポイントを以下に示す。

#### 2-1-2. 疑似コード

```typescript
// src/scraper/dedup.ts（Phase 3 で実装）

import { supabase } from '@/lib/supabase';
import { normalizeXUrl } from '@/lib/x-url-normalizer';

export async function upsertCandidate(candidate: ScrapedCandidate, seed: string) {
  const xUsername = normalizeXUrl(candidate.xLink);
  if (!xUsername) throw new Error('x_username 正規化失敗');

  // Phase 1 判定（AI絵師 / Pixivリンクなし → is_illustrator=false で INSERT）
  const isIllustrator = determineIsIllustrator(candidate);

  const { data, error } = await supabase
    .from('illustrators')
    .upsert(
      {
        x_username:        xUsername,
        display_name:      candidate.displayName,
        bio:               candidate.bio,
        follower_count:    candidate.followerCount,
        x_link:            candidate.xLink,
        pixiv_link:        candidate.pixivLink, // 初回のみ
        detected_from:     [seed],
        first_detected_at: new Date().toISOString(),
        last_seen_at:      new Date().toISOString(),
        is_illustrator:    isIllustrator, // null / true / false
      },
      {
        onConflict: 'x_username',
        // 既存行は last_seen_at と detected_from のみ更新する
        // アクティブカラムは触らない（Notion主導フィールドを潰さない）
        ignoreDuplicates: false,
      }
    )
    .select('id, is_illustrator, x_username');

  if (error) {
    await recordSyncFailure({
      source: 'scraper',
      target: 'supabase',
      operation: 'insert',
      error_message: error.message,
    });
    throw error;
  }
  return data;
}
```

#### 2-1-3. レート制限

- **Supabase**: REST API の制限は緩い（プランによる）。1秒あたり数十件の UPSERT は問題ない
- **スクレイパー側のボトルネック**が先に来る（X のレート制限、Browser Use の動作速度）
- バッチ単位で500件ずつ UPSERT することを推奨

#### 2-1-4. 失敗ハンドリング

- 個別レコードの失敗は `sync_failures` テーブルに記録（`source='scraper'`）
- バッチ全体の失敗は `scraping_logs.status='failed'` に記録
- リトライは同期ジョブと同じく最大10回・指数バックオフ

### 2-2. Supabase → Notion（10分おき）

#### 2-2-1. 概要

Supabase 側で変更があった `illustrators` レコードを検知して、Notion の該当ページに **Supabase 主導フィールド** を書き戻す。

**対象レコードの絞り込み**:

```sql
SELECT *
  FROM illustrators
 WHERE is_illustrator = TRUE
   AND (
        last_synced_to_notion_at IS NULL
     OR updated_at > last_synced_to_notion_at
   );
```

ポイント：
- `is_illustrator = TRUE` だけを同期（Notion に表示されるのは確定済みのみ）
- `last_synced_to_notion_at` が NULL（まだ一度も同期していない）、または `updated_at` の方が新しい場合のみ対象

**書き戻し対象フィールド**（Supabase 主導のみ）:

```
artist_name（作家名）
x_link（Xリンク）
follower_count（フォロワー数）
detected_from（検出元）
first_detected_at（初回検出日時）
last_seen_at（最終検出日時）
```

※ `display_name` や `bio` は Notion には現状プロパティがないので同期しない。必要になったら追加する。

#### 2-2-2. 新規レコードの扱い（Berryfeel 由来 or スクレイパー新規）

`notion_page_id IS NULL` のレコード（Notion メインDBにまだ存在しない）で、かつ `is_illustrator = TRUE` のものは **新規ページを作成**する。

- Berryfeel 別DB から統合したケース A のレコード
- 今後、スクレイパー経由でスカウトが確定させた新規レコード

新規作成時は全アクティブカラム（Notion主導フィールドも含む）を初期値としてセットし、作成した Notion ページの ID を Supabase 側の `notion_page_id` に保存する。

#### 2-2-3. 疑似コード

```typescript
// src/sync/supabase-to-notion.ts

import { supabase } from '@/lib/supabase';
import { notion } from '@/lib/notion';
import { logger } from '@/lib/logger';

const NOTION_MAIN_DB_ID = process.env.NOTION_MAIN_DB_ID!;

export async function syncSupabaseToNotion() {
  // 1. 同期対象を取得
  const { data: rows, error } = await supabase
    .from('illustrators')
    .select('*')
    .eq('is_illustrator', true)
    .or('last_synced_to_notion_at.is.null,updated_at.gt.last_synced_to_notion_at');

  if (error) throw error;
  logger.info({ count: rows?.length }, 'Supabase→Notion 同期対象');

  for (const row of rows ?? []) {
    try {
      if (row.notion_page_id) {
        // 既存ページ更新：Supabase主導フィールドのみ送る
        await notion.pages.update({
          page_id: row.notion_page_id,
          properties: buildSupabaseLedProperties(row),
        });
      } else {
        // 新規ページ作成：全フィールド送る
        const page = await notion.pages.create({
          parent: { database_id: NOTION_MAIN_DB_ID },
          properties: buildAllProperties(row),
        });
        // 作成したページIDを Supabase に保存
        await supabase
          .from('illustrators')
          .update({ notion_page_id: page.id })
          .eq('id', row.id);
      }

      // 2. 同期完了タイムスタンプを更新
      await supabase
        .from('illustrators')
        .update({ last_synced_to_notion_at: new Date().toISOString() })
        .eq('id', row.id);

      // 3. Notion API レート制限対策（3 req/sec）
      await sleep(400);
    } catch (e) {
      await recordSyncFailure({
        source: 'supabase',
        target: 'notion',
        record_id: row.id,
        operation: row.notion_page_id ? 'update' : 'insert',
        error_message: (e as Error).message,
      });
      logger.error({ err: e, id: row.id }, 'Supabase→Notion 同期失敗');
    }
  }
}

// Supabase主導フィールドのみを組み立てる
function buildSupabaseLedProperties(row: IllustratorRow) {
  return {
    '作家名':         { title: [{ text: { content: row.artist_name } }] },
    'Xリンク':        { url: row.x_link ?? null },
    'フォロワー数':   { number: row.follower_count ?? null },
    '検出元':         {
      multi_select: (row.detected_from ?? []).map((s) => ({ name: s })),
    },
    '初回検出日時':   { date: row.first_detected_at ? { start: row.first_detected_at } : null },
    '最終検出日時':   { date: row.last_seen_at ? { start: row.last_seen_at } : null },
  };
}

// 新規作成時は全フィールド（Notion主導含む）を初期値で送る
function buildAllProperties(row: IllustratorRow) {
  return {
    ...buildSupabaseLedProperties(row),
    'マスターステータス': { status: { name: row.master_status } },
    'ランク':             row.rank ? { select: { name: row.rank } } : { select: null },
    'オーナー確認':       {
      multi_select: (row.owner_confirmed_by ?? []).map((s) => ({ name: s })),
    },
    'ジャンル':           {
      multi_select: (row.genres ?? []).map((s) => ({ name: s })),
    },
    '絵柄タグ':           {
      multi_select: (row.style_tags ?? []).map((s) => ({ name: s })),
    },
    'Pixivリンク':        { url: row.pixiv_link ?? null },
    'ポートフォリオサイト': { url: row.portfolio_link ?? null },
    'その他連絡先':       { url: row.other_contact ?? null },
    'メール':             { email: row.email ?? null },
    'クレジット名義':     { rich_text: [{ text: { content: row.credit_name ?? '' } }] },
    '連絡した日':         row.contacted_at ? { date: { start: row.contacted_at } } : { date: null },
    '連絡した人_new':     {
      multi_select: (row.contacted_by ?? []).map((s) => ({ name: s })),
    },
    '備考':               { rich_text: [{ text: { content: row.note ?? '' } }] },
  };
}
```

#### 2-2-4. レート制限

- **Notion API**: 3 requests/sec（1ページあたり 400ms の sleep）
- 10分おきに対象件数が数百件に達した場合、1ジョブで数分かかる可能性あり
- タイムアウト対策：GitHub Actions の job 時間制限は十分広い（デフォルト6時間）

#### 2-2-5. 失敗ハンドリング

- `sync_failures` に `source='supabase', target='notion'` で記録
- 次回の10分サイクルで自動リトライ（更新対象は `updated_at > last_synced_to_notion_at` のままなので残り続ける）
- 10回連続失敗したレコードは `retry_count=10` で通知対象になる

### 2-3. Notion → Supabase（10分おき・ポーリング）

#### 2-3-1. 概要

Notion API の `last_edited_time` をポーリングして、前回同期時刻以降に編集されたページを Supabase に取り込む。

**対象ページの絞り込み**:

```typescript
// 前回同期時刻（sync_state テーブル or 最新 last_synced_from_notion_at から取得）
const lastSyncedAt = await getLastSyncedFromNotion();

// Notion API で last_edited_time > lastSyncedAt のページをクエリ
const pages = await notion.databases.query({
  database_id: NOTION_MAIN_DB_ID,
  filter: {
    timestamp: 'last_edited_time',
    last_edited_time: { after: lastSyncedAt },
  },
  sorts: [{ timestamp: 'last_edited_time', direction: 'ascending' }],
});
```

**反映対象フィールド**（Notion 主導のみ）:

```
master_status（マスターステータス）
rank（ランク）
owner_confirmed_by（オーナー確認）
style_tags（絵柄タグ）
genres（ジャンル）
note（備考）
contacted_at（連絡した日）
contacted_by（連絡した人_new）
email（メール）
portfolio_link（ポートフォリオサイト）
other_contact（その他連絡先）
credit_name（クレジット名義）
pixiv_link（Pixivリンク）※初回以降はNotion主導
```

**重要**: Supabase 主導フィールド（`artist_name`, `x_link`, `follower_count` 等）は **絶対に上書きしない**。これがパターンBの核心。

#### 2-3-2. Notion 側の新規ページ

Notion で新規作成されたページ（Supabase にない `page_id`）は Supabase に INSERT する。

```typescript
const existing = await supabase
  .from('illustrators')
  .select('id')
  .eq('notion_page_id', page.id)
  .maybeSingle();

if (!existing.data) {
  // Notion で新規作成されたページ → Supabase に INSERT
  await supabase.from('illustrators').insert({
    notion_page_id: page.id,
    artist_name: extractTitle(page),
    // Supabase主導フィールドも Notion にある値を初期値として入れる
    x_link: extractUrl(page, 'Xリンク'),
    x_username: normalizeXUrl(extractUrl(page, 'Xリンク')) ?? `(no-x-link-${page.id.slice(0, 16).replace(/-/g, '')})`,
    is_illustrator: true,
    ...extractNotionLedFields(page),
    last_synced_from_notion_at: new Date().toISOString(),
  });
}
```

`x_username` が空のケースはプレースホルダー `(no-x-link-<pageIdShort>)` を入れる（スキーマの NOT NULL + UNIQUE 制約を担保）。

#### 2-3-3. 削除・アーカイブされたページ

**現状の方針**: Supabase 側のレコードは **維持する**（削除しない）。

- Notion で手動削除されたページは稀だが、Supabase からも削除すると他のジャンル情報が失われる可能性がある
- 代わりに `illustrators.notion_page_id` を NULL にして「Notion とのリンクが切れた」状態にすることも検討可能だが、Phase 2 時点では実装保留
- 運用上、削除・アーカイブが必要になったら加藤さんが Supabase Studio で手動対応

#### 2-3-4. 疑似コード

```typescript
// src/sync/notion-to-supabase.ts

import { supabase } from '@/lib/supabase';
import { notion } from '@/lib/notion';
import { logger } from '@/lib/logger';

const NOTION_MAIN_DB_ID = process.env.NOTION_MAIN_DB_ID!;

export async function syncNotionToSupabase() {
  // 1. 前回の取り込み時刻（sync_state テーブルから取得）
  const lastSyncedAt = await getLastSyncedFromNotion();

  // 2. Notion API でページをページネーション取得
  let cursor: string | undefined = undefined;
  const allPages: any[] = [];
  do {
    const res = await notion.databases.query({
      database_id: NOTION_MAIN_DB_ID,
      filter: {
        timestamp: 'last_edited_time',
        last_edited_time: { after: lastSyncedAt },
      },
      sorts: [{ timestamp: 'last_edited_time', direction: 'ascending' }],
      start_cursor: cursor,
      page_size: 100,
    });
    allPages.push(...res.results);
    cursor = res.has_more ? res.next_cursor ?? undefined : undefined;
    await sleep(400); // レート制限対策
  } while (cursor);

  logger.info({ count: allPages.length }, 'Notion→Supabase 同期対象');

  // 3. 各ページを Supabase に反映
  for (const page of allPages) {
    try {
      const { data: existing } = await supabase
        .from('illustrators')
        .select('id, last_synced_from_notion_at, last_synced_to_notion_at')
        .eq('notion_page_id', page.id)
        .maybeSingle();

      if (existing) {
        // 既存ページ：Notion主導フィールドのみ更新
        await supabase
          .from('illustrators')
          .update({
            ...extractNotionLedFields(page),
            last_synced_from_notion_at: new Date().toISOString(),
          })
          .eq('id', existing.id);
      } else {
        // Notion で新規作成されたページ → Supabase に INSERT
        await supabase.from('illustrators').insert({
          notion_page_id: page.id,
          artist_name: extractTitle(page) ?? `(名無し-${page.id.slice(0, 16).replace(/-/g, '')})`,
          x_link: extractUrl(page, 'Xリンク'),
          x_username:
            normalizeXUrl(extractUrl(page, 'Xリンク')) ??
            `(no-x-link-${page.id.slice(0, 16).replace(/-/g, '')})`,
          is_illustrator: true,
          ...extractNotionLedFields(page),
          last_synced_from_notion_at: new Date().toISOString(),
        });
      }
    } catch (e) {
      await recordSyncFailure({
        source: 'notion',
        target: 'supabase',
        operation: 'update',
        error_message: (e as Error).message,
      });
      logger.error({ err: e, page_id: page.id }, 'Notion→Supabase 同期失敗');
    }
  }

  // 4. 全件成功したら sync_state の last_synced_from_notion_at を今回実行の開始時刻に更新
  // （ページ毎の last_synced_from_notion_at とは別に、グローバル値を sync_state に持つ）
  await updateSyncState('notion_to_supabase', new Date().toISOString());
}

function extractNotionLedFields(page: any) {
  const p = page.properties;
  return {
    master_status:      p['マスターステータス']?.status?.name ?? '候補',
    rank:               p['ランク']?.select?.name ?? null,
    owner_confirmed_by: p['オーナー確認']?.multi_select?.map((s: any) => s.name) ?? [],
    style_tags:         p['絵柄タグ']?.multi_select?.map((s: any) => s.name) ?? [],
    genres:             p['ジャンル']?.multi_select?.map((s: any) => s.name) ?? [],
    note:               p['備考']?.rich_text?.map((r: any) => r.plain_text).join('') ?? '',
    contacted_at:       p['連絡した日']?.date?.start ?? null,
    contacted_by:       p['連絡した人_new']?.multi_select?.map((s: any) => s.name) ?? [],
    email:              p['メール']?.email ?? null,
    portfolio_link:     p['ポートフォリオサイト']?.url ?? null,
    other_contact:      p['その他連絡先']?.url ?? null,
    credit_name:        p['クレジット名義']?.rich_text?.map((r: any) => r.plain_text).join('') ?? '',
    pixiv_link:         p['Pixivリンク']?.url ?? null,
  };
}
```

#### 2-3-5. レート制限

- **Notion API**: 3 requests/sec
- データベースクエリは1回あたり最大100件取得。数百件ヒットするとページネーションで数秒〜10秒
- 取得と書き込みで分けて、取得 → ループ書き込みの構造にする

#### 2-3-6. 失敗ハンドリング

- `sync_failures` に `source='notion', target='supabase'` で記録
- 同期元の `last_synced_from_notion_at`（sync_state テーブル）は **全件成功時のみ更新**
- 部分失敗時は次回同じ時刻から再取得するため、失敗レコードだけ再試行されず **全体を再取得** することになる（冪等なので問題ない）

### 2-4. Supabase → Google Sheets（10分おき）

#### 2-4-1. 概要

候補プール（`is_illustrator IS NULL` のレコード）を Google Sheets に表示する。スカウトがシート上で仮ランク付け・判定を行う。

**対象レコード**:

```sql
SELECT *
  FROM illustrators
 WHERE is_illustrator IS NULL
   AND (
        last_synced_to_sheet_at IS NULL
     OR updated_at > last_synced_to_sheet_at
   )
 ORDER BY first_detected_at DESC;
```

- `is_illustrator IS NULL` のみ（未判定のみ Sheets に表示）
- スカウトが判定して `is_illustrator` が `true`/`false` になると自動的に Sheets から消える（次サイクルで）

**書き込む列**（Sheets の A〜I 列、自動入力分）:

| 列 | ヘッダー | Supabaseのソース |
|---|---|---|
| A | 検出日 | `first_detected_at` |
| B | 検出元 | `detected_from[0]`（複数あれば全て結合） |
| C | Xアカウント | `x_username` |
| D | 表示名 | `display_name` |
| E | プロフィール | `bio`（最大500文字） |
| F | フォロワー数 | `follower_count` |
| G | Pixivリンク | `pixiv_link` |
| H | ポートフォリオ | `portfolio_link` |
| I | 既存DB重複 | 常に `NO`（重複はSupabase側で除外済み） |

**J, K, L 列（スカウト入力）は上書きしない**。M 列（同期状態）は `同期済み` に更新。

#### 2-4-2. Phase 3 依存の補足

合意事項リスト上、Sheets 候補プールに新規候補が流入するのは **Phase 3（スクレイパー実装）が稼働してから**。Phase 2 の時点では：

- スクレイパーが動いていないので `is_illustrator IS NULL` の新規レコードはほぼ存在しない
- この同期ジョブは**空で回る**（0件処理）
- Phase 3 稼働後に自然に流入が始まる

ただし **ジョブ自体は Phase 2 で実装・常時稼働** させておく（Phase 3 でのテスト用）。

#### 2-4-3. 疑似コード

```typescript
// src/sync/supabase-to-sheet.ts

import { supabase } from '@/lib/supabase';
import { sheets } from '@/lib/sheets';
import { logger } from '@/lib/logger';

const SHEET_ID = process.env.GOOGLE_SHEET_ID!;
const SHEET_RANGE = '候補プール!A2:M'; // ヘッダー行は除く

export async function syncSupabaseToSheet() {
  const { data: rows, error } = await supabase
    .from('illustrators')
    .select('*')
    .is('is_illustrator', null)
    .or('last_synced_to_sheet_at.is.null,updated_at.gt.last_synced_to_sheet_at')
    .order('first_detected_at', { ascending: false });

  if (error) throw error;
  logger.info({ count: rows?.length }, 'Supabase→Sheet 同期対象');

  for (const row of rows ?? []) {
    try {
      // row.sheet_row_index があれば UPDATE、なければ APPEND
      if (row.sheet_row_index) {
        // 既存行の A〜I 列のみ更新（J〜L はスカウト入力を保護）
        await sheets.spreadsheets.values.update({
          spreadsheetId: SHEET_ID,
          range: `候補プール!A${row.sheet_row_index}:I${row.sheet_row_index}`,
          valueInputOption: 'RAW',
          requestBody: { values: [toRowA2I(row)] },
        });
      } else {
        // APPEND で追加
        const appendRes = await sheets.spreadsheets.values.append({
          spreadsheetId: SHEET_ID,
          range: SHEET_RANGE,
          valueInputOption: 'RAW',
          insertDataOption: 'INSERT_ROWS',
          requestBody: { values: [toRowFull(row)] }, // A〜M 全列
        });
        // 追加された行番号を取得して保存
        const updatedRange = appendRes.data.updates?.updatedRange ?? '';
        const rowIndex = parseRowIndex(updatedRange); // 例: '候補プール!A15:M15' → 15
        await supabase
          .from('illustrators')
          .update({
            sheet_row_index: rowIndex,
            last_synced_to_sheet_at: new Date().toISOString(),
          })
          .eq('id', row.id);
      }

      // last_synced_to_sheet_at を更新
      await supabase
        .from('illustrators')
        .update({ last_synced_to_sheet_at: new Date().toISOString() })
        .eq('id', row.id);
    } catch (e) {
      await recordSyncFailure({
        source: 'supabase',
        target: 'sheets',
        record_id: row.id,
        operation: row.sheet_row_index ? 'update' : 'insert',
        error_message: (e as Error).message,
      });
    }
  }
}

function toRowA2I(row: IllustratorRow): any[] {
  return [
    row.first_detected_at?.slice(0, 10),       // A: 検出日
    (row.detected_from ?? []).join(', '),      // B: 検出元
    row.x_username,                            // C: Xアカウント
    row.display_name,                          // D: 表示名
    (row.bio ?? '').slice(0, 500),             // E: プロフィール
    row.follower_count,                        // F: フォロワー数
    row.pixiv_link,                            // G: Pixivリンク
    row.portfolio_link,                        // H: ポートフォリオ
    'NO',                                      // I: 既存DB重複
  ];
}

function toRowFull(row: IllustratorRow): any[] {
  return [...toRowA2I(row), '', '', '', '未同期']; // J,K,L（空）, M=未同期
}
```

#### 2-4-4. レート制限

- **Google Sheets API**: 60 requests/min/user（読み取り）、60 requests/min/user（書き込み）
- バッチ更新を使うとレート制限を消費せずに複数行を一括更新できる
- 数百件を超える場合は `batchUpdate` を使うよう最適化

#### 2-4-5. 失敗ハンドリング

- `sync_failures` に `source='supabase', target='sheets'` で記録
- 次サイクルでリトライ

### 2-5. Google Sheets → Supabase（10分おき）

#### 2-5-1. 概要

スカウトが Sheets の J・K・L 列に入力した内容を Supabase に反映する。

**対象行の検知**:
- Sheets 側では「変更検知」が難しいため、**全行を取得して M 列（同期状態）=`未同期` の行だけを処理**する
- 反映後、M 列を `同期済み` に更新

**反映対象フィールド**:

| Sheets列 | Supabaseフィールド | 説明 |
|---|---|---|
| J（判定） | `is_illustrator` | 空 or `イラストレーターじゃない`<br>→ 空なら `NULL`、「イラストレーターじゃない」なら `false` |
| K（仮ランク） | `rank` | `S`/`A`/`B`/`C` が入力されたら反映<br>入力があれば同時に `is_illustrator = true` も設定 |
| L（コメント） | `scout_comment` | 自由記述 |

ロジック：
- **K 列に仮ランクが入力された** → `is_illustrator = true` + `rank` を設定（Notion に流れる）
- **J 列に「イラストレーターじゃない」が入力された** → `is_illustrator = false`
- **どちらも空** → `is_illustrator = NULL`（未判定、次回も Sheets に残る）

#### 2-5-2. 疑似コード

```typescript
// src/sync/sheet-to-supabase.ts

import { supabase } from '@/lib/supabase';
import { sheets } from '@/lib/sheets';
import { logger } from '@/lib/logger';

const SHEET_ID = process.env.GOOGLE_SHEET_ID!;

export async function syncSheetToSupabase() {
  // 1. 候補プールシートを全件取得
  const res = await sheets.spreadsheets.values.get({
    spreadsheetId: SHEET_ID,
    range: '候補プール!A2:M',
  });
  const allRows = res.data.values ?? [];
  logger.info({ count: allRows.length }, 'Sheet→Supabase 取得行数');

  const updates: any[] = [];

  for (let i = 0; i < allRows.length; i++) {
    const [
      detectedAt, detectedFrom, xUsername, displayName, bio,
      followerCount, pixivLink, portfolio, isDup,
      judgment, tentativeRank, scoutComment, syncStatus,
    ] = allRows[i];

    // M 列が '同期済み' ならスキップ
    if (syncStatus === '同期済み') continue;
    // スカウトが何も入力していないスキップ
    if (!judgment && !tentativeRank && !scoutComment) continue;

    const rowIndex = i + 2; // 1-indexed + ヘッダー行

    try {
      // Supabase 側を探す（x_username で紐付け）
      const { data: target } = await supabase
        .from('illustrators')
        .select('id')
        .eq('x_username', xUsername)
        .maybeSingle();

      if (!target) {
        await recordSyncFailure({
          source: 'sheets',
          target: 'supabase',
          operation: 'update',
          error_message: `x_username=${xUsername} が Supabase に見つかりません（行 ${rowIndex}）`,
        });
        continue;
      }

      // 更新内容を組み立てる
      const patch: any = {
        scout_comment: scoutComment || null,
        last_synced_from_sheet_at: new Date().toISOString(),
      };

      if (judgment === 'イラストレーターじゃない') {
        patch.is_illustrator = false;
      } else if (tentativeRank && ['S', 'A', 'B', 'C'].includes(tentativeRank)) {
        patch.is_illustrator = true;
        patch.rank = tentativeRank;
      }
      // 両方空なら NULL のまま（未判定維持）

      await supabase
        .from('illustrators')
        .update(patch)
        .eq('id', target.id);

      // M 列を '同期済み' に更新するリストに追加
      updates.push({
        range: `候補プール!M${rowIndex}`,
        values: [['同期済み']],
      });
    } catch (e) {
      await recordSyncFailure({
        source: 'sheets',
        target: 'supabase',
        operation: 'update',
        error_message: (e as Error).message,
      });

      // M 列を '同期失敗' に
      updates.push({
        range: `候補プール!M${rowIndex}`,
        values: [['同期失敗']],
      });
    }
  }

  // 2. M 列を一括更新
  if (updates.length > 0) {
    await sheets.spreadsheets.values.batchUpdate({
      spreadsheetId: SHEET_ID,
      requestBody: {
        valueInputOption: 'RAW',
        data: updates,
      },
    });
  }
}
```

#### 2-5-3. レート制限

- **Google Sheets API**: 60 requests/min/user
- 全行取得は1リクエストで済むが、`batchUpdate` で複数行一括更新することが望ましい

#### 2-5-4. 失敗ハンドリング

- `sync_failures` に `source='sheets', target='supabase'` で記録
- M 列を `同期失敗` にしておけば、次サイクルで再トライされる（同期済みになるまで残る）

## 4. 失敗ハンドリング

### 4.1 sync_failures テーブルへの記録

全ての同期失敗は `sync_failures` に蓄積する。

```typescript
// src/lib/sync-failure.ts

export async function recordSyncFailure(params: {
  source: 'supabase' | 'notion' | 'sheets' | 'scraper';
  target: 'supabase' | 'notion' | 'sheets';
  record_id?: string;
  operation: 'insert' | 'update' | 'delete' | 'fetch';
  error_message: string;
}) {
  await supabase.from('sync_failures').insert({
    source: params.source,
    target: params.target,
    record_id: params.record_id ?? null,
    operation: params.operation,
    error_message: params.error_message,
    retry_count: 0,
  });
}
```

### 4.2 自動リトライ

- **最大10回、指数バックオフ**
- 再試行は「次の10分サイクル」で発生する（同期対象に `updated_at > last_synced_to_*_at` がかかっているため、失敗レコードは自動的に再選択される）
- リトライ回数は sync_failures.retry_count に記録
- 実装方針：

```typescript
// src/sync/retry-failed.ts

export async function retryFailed() {
  // 未解決の失敗レコードを取得
  const { data: failures } = await supabase
    .from('sync_failures')
    .select('*')
    .is('resolved_at', null)
    .lt('retry_count', 10);

  for (const f of failures ?? []) {
    // 指数バックオフ（最低5分・最大24時間）
    const backoffMin = Math.min(5 * 2 ** f.retry_count, 60 * 24);
    const nextRetryAt = new Date(new Date(f.created_at).getTime() + backoffMin * 60 * 1000);
    if (new Date() < nextRetryAt) continue;

    try {
      // ジョブ別にリトライロジックを呼ぶ
      await retryByJob(f);
      await supabase
        .from('sync_failures')
        .update({ resolved_at: new Date().toISOString() })
        .eq('id', f.id);
    } catch {
      await supabase
        .from('sync_failures')
        .update({ retry_count: f.retry_count + 1 })
        .eq('id', f.id);
    }
  }
}
```

### 4.3 リトライ10回失敗後

- `retry_count = 10` に達したレコードは **通知対象**
- 以降は自動リトライを止める
- 加藤さんが Notion ログを見て手動対応 or Supabase Studio で `resolved_at` を手動設定して解決扱いにする

## 5. 失敗通知

### 5.1 Notion ログページへの追記

合意事項リスト v2.2 で **メール通知は廃止、Notion ログページに統一**と決定済み。

- ページID：`.env.local` の `NOTION_SYNC_LOG_PAGE_ID`
- 1時間おきに失敗0件でなければ追記ブロックを追加
- 通知フォーマット：timestamp、source、target、件数、エラー概要

### 5.2 疑似コード

```typescript
// src/lib/notion-sync-log.ts

import { notion } from '@/lib/notion';
import { supabase } from '@/lib/supabase';
import { logger } from '@/lib/logger';

const LOG_PAGE_ID = process.env.NOTION_SYNC_LOG_PAGE_ID!;

export async function notifyFailures() {
  // retry_count >= 10 かつ resolved_at IS NULL の未解決失敗を集計
  const { data: failures } = await supabase
    .from('sync_failures')
    .select('source, target, error_message, created_at')
    .is('resolved_at', null)
    .gte('retry_count', 10);

  if (!failures || failures.length === 0) {
    logger.info('未解決の失敗なし。通知スキップ');
    return;
  }

  // source×target で集計
  const grouped: Record<string, typeof failures> = {};
  for (const f of failures) {
    const key = `${f.source}→${f.target}`;
    (grouped[key] ??= []).push(f);
  }

  // Notion ログページに追記
  const now = new Date().toISOString().replace('T', ' ').slice(0, 16);
  const children: any[] = [
    {
      object: 'block',
      type: 'heading_3',
      heading_3: {
        rich_text: [{ text: { content: `[${now}] 同期失敗通知（計 ${failures.length} 件）` } }],
      },
    },
  ];

  for (const [key, items] of Object.entries(grouped)) {
    children.push({
      object: 'block',
      type: 'bulleted_list_item',
      bulleted_list_item: {
        rich_text: [
          {
            text: {
              content: `${key}: ${items.length} 件 / 例: ${items[0].error_message.slice(0, 200)}`,
            },
          },
        ],
      },
    });
  }

  try {
    await notion.blocks.children.append({
      block_id: LOG_PAGE_ID,
      children,
    });
    logger.info({ count: failures.length }, '失敗通知を Notion に記録');
  } catch (e) {
    // Notion に送れない場合は pino で stderr に出力（フォールバック）
    logger.error({ err: e, failures }, '失敗通知の Notion 追記に失敗。stderr にフォールバック');
  }
}
```

### 5.3 実行タイミング

- GitHub Actions の cron: `'0 * * * *'`（毎時0分）
- 失敗0件ならスキップ（Notion に書き込まない）

### 5.4 Notion に送れない場合のフォールバック

- pino で stderr に出力
- GitHub Actions のログに残るので、加藤さんが Actions 画面で確認可能

## 6. ループ防止

### 6.1 書き込み後のタイムスタンプ更新

同期ジョブは **書き込み後に必ず `last_synced_*` を更新**することで、次サイクルでは自分の書き込みを拾わない。

| 方向 | 書き込み後に更新するフィールド |
|---|---|
| Supabase → Notion | `illustrators.last_synced_to_notion_at = NOW()` |
| Notion → Supabase | `illustrators.last_synced_from_notion_at = NOW()` |
| Supabase → Sheets | `illustrators.last_synced_to_sheet_at = NOW()` |
| Sheets → Supabase | `illustrators.last_synced_from_sheet_at = NOW()` + シートM列を `同期済み` |

### 6.2 フィールド別主体ルール

パターンB の本質は **同じフィールドを両側で編集しない** こと。これにより、双方向の書き込みがぶつからない。

- **Supabase → Notion**: Supabase主導フィールドだけを送る
  → Notion側でユーザーが編集した値（master_status 等）は**絶対に上書きしない**
- **Notion → Supabase**: Notion主導フィールドだけを取り込む
  → Supabase側のスクレイパー由来の値（artist_name, follower_count 等）は**絶対に上書きしない**

### 6.3 同一フィールド両側変更の競合

原則、発生しない設計だが、万一発生した場合（例：オーナーが Notion で `artist_name` を手書きしたケース）：

- **デフォルトルール**: Notion の `last_edited_time` が新しい方を勝たせる
- 具体的には Notion→Supabase 取り込み時、本来は Notion主導フィールドのみ反映するが、「Notion で明示的に変更された Supabase主導フィールド」があれば個別判断で取り込む（Phase 2 時点では自動化せず、Notion ログに警告を残すだけ）

### 6.4 updated_at との関係

- **`updated_at` はトリガーで自動更新**される（どんな書き込みでも）
- 同期ジョブで書き込んだ直後に `updated_at` も更新されるため、次サイクルで `updated_at > last_synced_to_notion_at` が **成立**してしまう可能性がある
- これを回避するため、**同期ジョブは書き込み後に `last_synced_to_*_at = NOW()` を `updated_at` と同じタイミングで更新**する
- クエリでは `updated_at > last_synced_to_notion_at` を使うので、両者が同時刻なら同期対象から外れる

```
    ├── sync-all.yml            # 10分おき（Supabase↔Notion, Supabase↔Sheets）
    ├── auto-transition.yml     # 1日1回（連絡中→返信なし 30日遷移）
    └── notify-failures.yml     # 1時間おき（Notion ログページへ通知）
```

### 7.2 実装方針

- **TypeScript ESM**（`"type": "module"`、`tsconfig.json` で `module: NodeNext`）
- **Node.js v22 以上**（GitHub Actions では `actions/setup-node@v4` で指定）
- **依存関係**:
  - `@supabase/supabase-js`
  - `@notionhq/client`
  - `googleapis`
  - `pino` + `pino-pretty`
- **実行コマンド**: `pnpm tsx src/sync/xxx.ts`
- **エラー時は必ず `sync_failures` に記録してから throw する**（握りつぶしNG）
- **すべての操作に pino ログを必ず出す**（GitHub Actions で追跡可能にする）

### 7.3 sync_state テーブル（新規追加提案）

`last_synced_from_notion_at` のような「ジョブ全体のグローバル値」が必要になる。各レコードに対してはすでに同名カラムを持っているが、**ジョブ単位の最終成功時刻**を記録する用途で `sync_state` テーブルを追加するのを推奨。

```sql
CREATE TABLE sync_state (
  job_name     TEXT PRIMARY KEY,  -- 'notion_to_supabase', 'supabase_to_notion' 等
  last_run_at  TIMESTAMPTZ NOT NULL,
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Phase 2 実装時に必要に応じて追加マイグレーションを行う（本スキーマ書 v1.1 には含まれていない）。

## 9. 失敗通知用 Notion ログページの設計

### 9.1 ページ構成

- **Notion ワークスペース内にフラットな1ページ**を用意
- タイトル例：「同期失敗ログ」
- 作成したら URL から `page_id`（32文字のハイフン入りID）を取得し、`.env.local` に登録
- GitHub Secrets にも同じ値を `NOTION_SYNC_LOG_PAGE_ID` として登録

### 9.2 ブロック構造

1時間おきに以下のブロックを append する（既存は削除しない、追記のみ）：

```
H3 ブロック: [2026-04-23 10:00] 同期失敗通知（計 3 件）
  bullet: supabase→notion: 2 件 / 例: Notion API 400 Bad Request...
  bullet: sheets→supabase: 1 件 / 例: x_username=xxx が見つかりません...

H3 ブロック: [2026-04-23 11:00] 同期失敗通知（計 1 件）
  bullet: notion→supabase: 1 件 / 例: ...
```

### 9.3 ログクリーンアップ

- 過去ログは **折り畳み or 定期クリーンアップ**
- 運用 1〜3ヶ月後を目安に、古いログブロックを手動削除 or 別ページにアーカイブ
- 自動化は Phase 2 では実装しない（必要に応じて Phase 5 で検討）

## 10. 運用上の注意

### 10.1 マイグレーション完了から同期ジョブ稼働までの注意

- マイグレーション直後は **同期ジョブが動き始めるまでの間、Notion 編集を控える**よう北條・三村さんに周知する
- 同期ジョブ稼働の順番：
  1. マイグレーション完了（03_マイグレーション手順.md）
  2. Step 11 で GitHub Actions を有効化
  3. 最初の10分サイクルで Supabase → Notion の初回フル同期（全レコードが対象）
  4. 最初のサイクルが完了したら通常運用開始

### 10.2 シークレット管理（GitHub Secrets）

Phase 2 で必要なシークレット一覧：

| キー名 | 用途 |
|---|---|
| `SUPABASE_URL` | Supabase プロジェクト URL |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase service_role キー（RLS バイパス） |
| `NOTION_API_KEY` | Notion インテグレーション `Claude.Bisque` のシークレット |
| `NOTION_MAIN_DB_ID` | Notion メインDBのID |
| `NOTION_SYNC_LOG_PAGE_ID` | 失敗通知ログページのID |
| `GOOGLE_SHEET_ID` | スカウト作業用スプレッドシートID |
| `GOOGLE_SERVICE_ACCOUNT_JSON` | Google サービスアカウント鍵 JSON（文字列） |

### 10.3 Notion 側で新規ページ作成時の注意

Notion → Supabase で新規ページを Supabase に INSERT する場合、Notion 側に **以下の選択肢が事前に存在**していないと取り込み時に不整合が起きる：

- マスターステータス（7値）
- 連絡した人_new（multi_select、拡張可能）
- オーナー確認（3値）
- 絵柄タグ（4値）
- ジャンル（6値）
- ポートフォリオサイト（url）

選択肢は Step 1 のマイグレーション準備で揃えている前提。

### 10.4 同期ジョブの監視

- GitHub Actions 画面で直近の実行を確認（`https://github.com/takuyakato/illustrator-scraper/actions`）
- **3連続で失敗0件** が運用開始の判断基準
- それ以降も、失敗が10件以上連続したら Notion ログページで要因を確認

### 10.5 Phase 3 稼働時の再確認

Phase 3（スクレイパー）が稼働を始めると、`is_illustrator = NULL` のレコードが大量に追加される。Supabase → Sheets 同期の件数が急増するので、以下を確認：

- Google Sheets API のレート制限に抵触しないか（1ジョブ数百件が目安）
- スクレイパーの1バッチ実行で1000件を超える場合は、同期ジョブを分割実行することを検討
