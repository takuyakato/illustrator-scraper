# 00. 実装ロードマップ

**バージョン**: 1.0
**作成日**: 2026-04-22
**目的**: 仕様確定後の実装作業を Phase 0〜4 に分け、作業順序・依存関係・前提条件を一覧化する

このドキュメントは **「次に何から実装するか」を迷わずに進めるためのガイド** です。
仕様の詳細は [合意事項リスト v2.1](../00_合意事項リスト.md) を参照。

---

## 📐 全体の実装順序

```
Phase 0: 基盤準備
   ↓（Supabase・Notion・GitHub の初期設定）
Phase 1: マイグレーション
   ↓（Notion → Supabase への初回データ移行）
Phase 2: 同期ジョブ実装
   ↓（Supabase ↔ Notion、Supabase ↔ Sheets の双方向同期）
Phase 3: スクレイパー実装
   ↓（X フォロー取得・Pixivリンク判定・重複チェック）
Phase 4: 運用開始
   ↓（初回フルスキャン・オーナー初期作業・チーム周知）
Phase 5: 拡張（Phase 2以降の機能強化）
```

各 Phase は **前の Phase が完了してから着手** する。並列化は不可（後続 Phase は前 Phase の成果物に依存）。

---

## 📋 Phase 1: マイグレーション

### 目的

既存の Notion データ（メイン DB + Berryfeel 別 DB）を Supabase に統合移行し、同時にスキーマに合わせてクリーンアップする。

### 前提条件

- Phase 0 完了
- Supabase スキーマが適用済みで、illustrators テーブルが空

### 作業一覧

[`docs/architecture/03_マイグレーション手順.md` v1.1 の Step 0〜12 に従う](./03_マイグレーション手順.md)

Summary：

1. Step 0: バックアップ（Notion CSV、Supabase schema dump）
2. Step 1: Notion 新プロパティ追加（マスターステータス、オーナー確認、絵柄タグ、ポートフォリオサイト、連絡した人_new）
3. Step 2: マイグレーションスクリプト `02_notion_to_supabase.ts` 実行
4. Step 3〜8: 変換・統合（Step 2スクリプト内で実施）
5. Step 9: 変換結果のオーナー目視レビュー
6. Step 10: Notion 側クリーンアップ（マスターステータス書き戻し、旧プロパティ非表示）
7. Step 11: 同期ジョブ起動準備（Phase 2完了後）
8. Step 12: 疎通確認

### 完了条件

`docs/architecture/03_マイグレーション手順.md` Section 10「初期運用開始の判断基準」を満たす：

- [ ] 全レコードが `illustrators` テーブルに存在（Notion と件数一致）
- [ ] `rank` / `master_status` / `contacted_by` が適切に設定されている
- [ ] `owner_confirmed_by` が全員空配列（要確認）
- [ ] Berryfeel 別 DB が `[ARCHIVED]` リネーム済み
- [ ] Notion 側の不要プロパティが非表示化されている

### 所要時間目安

- **半日〜1日**

## 📋 Phase 2: 同期ジョブ実装

### 目的

Supabase ↔ Notion・Supabase ↔ Google Sheets の双方向同期を実装する。

### 前提条件

- Phase 1 完了
- Supabase にマイグレーション済みデータが存在

### 作業一覧

#### 2-1. 同期ロジック技術仕様書を作成

- [ ] `docs/architecture/04_同期ロジック.md` を新規作成
  - 5つの同期ジョブの詳細（S2N・N2S・S2Sh・Sh2S・自動遷移）

  - ループ防止ロジック（`updated_at > last_synced_from_*`）
  - 失敗ハンドリング（sync_failures テーブル・最大10回リトライ）

- [ ] `tsconfig.json` を作成（ESM、target ES2022）

- [ ] Dev 依存関係：`typescript`、`tsx`、`eslint`、`prettier`、`vitest`

#### 2-3. 共通ライブラリ実装

- [ ] `src/lib/supabase.ts`（Supabase クライアント・型定義）
- [ ] `src/lib/notion.ts`（Notion クライアント・プロパティ変換）
- [ ] `src/lib/sheets.ts`（Google Sheets クライアント）
- [ ] `src/lib/x-url-normalizer.ts`（Xリンク正規化）
- [ ] `src/lib/portfolio-splitter.ts`（ポートフォリオ振り分け）
- [ ] `src/lib/status-mapper.ts`（マスターステータス変換）
- [ ] `src/lib/mailer.ts`（失敗通知メール送信）
- [ ] `src/lib/logger.ts`（pino 設定）

#### 2-4. 同期ジョブ実装

- [ ] `src/jobs/sync-supabase-to-notion.ts`（S2N）
- [ ] `src/jobs/sync-notion-to-supabase.ts`（N2S・ポーリング方式）

- [ ] `src/jobs/sync-sheet-to-supabase.ts`（Sh2S）
- [ ] `src/jobs/auto-transition.ts`（連絡中→返信なし 14日経過）

#### 2-6. ユニットテスト実装

- [ ] Xリンク正規化のテスト（各パターン網羅）
- [ ] ポートフォリオ振り分けのテスト
- [ ] マスターステータス変換のテスト

- [ ] 10分おきに全同期ジョブが正常実行される

## 🚨 フェーズ移行時の注意

### 各フェーズ終了時の確認

- 完了条件を全て満たしているか
- チェックリスト項目に未完があれば残タスクとして記録

### 失敗時の対応

- 実装中に問題発生 → `docs/architecture/99_整合性チェックリスト.md` に記録
- 仕様変更が必要 → 合意事項リストを v2.x にアップデート、関連ドキュメントも揃えて更新

### ドキュメントバージョン管理

- 合意事項リスト：仕様変更のたびに v2.x にバンプ
- スキーマ書・マイグレーション手順書：v1.x でバンプ
- 変更履歴を必ず改訂履歴に記録
