## 🚨 実装時に特に注意すべき落とし穴

### 1. Notion status型プロパティの事前作成

`status` 型プロパティは、**事前に全ての選択肢をNotion UIで作成**してから API で値をセットしないと 400 エラーになる。

- マスターステータスの7値（候補/連絡中/返信なし/多忙辞退/条件次第/依頼成功/依頼不可）を Step 1 で全て作成する
- グループ（To-do/In progress/Complete）も設定する

### 2. multi_select の既存データ移行

Notion の `select` → `multi_select` への型変更は、データ損失の可能性がある。
- 既存「連絡した人」は `select` なので、新プロパティ「連絡した人_new」を `multi_select` で作成
- データを移行してから、旧プロパティを非表示にする（削除しない）

### 3. Notion API レート制限

- 3 requests/sec の制限
- 数百件〜数千件のバルク処理では **400ms の sleep** を挟む
- バルク処理には時間がかかる（1000件で約7分）

### 4. Xリンク正規化の落とし穴

- Notion上のXリンクには `https://`, `http://`, プロトコルなし、クエリパラメータ付き、末尾スラッシュあり、など複数パターンがある
- マイグレーション時に全パターンを正規化する関数 `normalize_x_username` を使う
- 正規化後、UNIQUE制約違反が発生した場合は手動で重複を解消する必要がある

### 5. Supabase ENUM の追加・変更

- `ALTER TYPE ... ADD VALUE` は**トランザクション内で実行できない**
- マイグレーションファイルを分割する必要がある

### 6. 循環同期（無限ループ）の防止

- Supabase→Notion 同期でNotionに書き込み → Notion側の `last_edited_time` が更新 → Notion→Supabase 同期で取り込み → ループ
- 防止策：`updated_at > last_synced_from_notion_at` かつ **Supabase主導フィールドに変更あり** のときのみNotionへ送る
- 同様に Notion→Supabase 側も同期元フィールドのみ更新対象とする
