# 06. 同期障害の再発防止設計

**作成日**: 2026-05-06
**対象**: `illustrators` の同期・migration 運用

## 1. 今回の障害から見直したこと

今回の障害は、`illustrators` のカラム削除と `update_updated_at()` トリガー関数の参照更新が分離していたことが原因。

`update_updated_at()` は同期ループを防ぐために「同期メタデータだけの UPDATE では `updated_at` を進めない」役割を持っていたが、実データカラムを関数内に手書きで列挙していた。そのため、削除済みカラムへの `NEW.*` / `OLD.*` 参照が残り、すべての UPDATE が失敗した。

見直し後の方針は、**カラム追加・削除に追従が必要な設計をやめる**こと。

## 2. 恒久対策

`update_updated_at()` は `to_jsonb(NEW)` / `to_jsonb(OLD)` の動的比較方式に変更する。

除外するのは以下の同期メタデータだけ。

- `last_synced_to_notion_at`
- `last_synced_from_notion_at`
- `last_synced_to_sheet_at`
- `last_synced_from_sheet_at`
- `updated_at`

この5つだけが変わった UPDATE では `updated_at` を据え置く。それ以外の値が変わった UPDATE では `updated_at = NOW()` にする。

この設計なら、`illustrators` に通常の実データカラムを追加・削除しても、トリガー関数のカラム列挙を更新する必要がない。

## 3. 同期設計の整理

同期ループ防止は次の責務分担にする。

| 層 | 責務 |
|---|---|
| DBトリガー | 実データ変更だけ `updated_at` を進める |
| 同期ジョブ | 外部サービスへの書き込み成功後に `last_synced_*_at` を単独更新する |
| pending VIEW | `updated_at > last_synced_to_*_at` で次回同期対象を抽出する |
| `sync_failures` | 個別失敗を `failure_key` で畳み込み、発生回数・最終発生時刻・解決済み状態で監視する |

同期ジョブ側で「`updated_at` と同じタイミングにする」ことへ依存しない。DBトリガー側で同期メタデータ単独更新を明示的に除外する。

## 4. migration 運用ルール

今後のスキーマ変更では、以下を必ず見る。

- `supabase/migrations/` に単独 migration を追加する
- 初期構築用の `supabase/_apply_all.sql` にも追記する
- `docs/architecture/02_Supabaseスキーマ.md` の現在仕様を更新する
- `docs/architecture/04_同期ロジック.md` の同期前提が変わらないか確認する
- TypeScript の `src/lib/types.ts` と Notion / Sheets 変換処理に影響がないか確認する
- `DROP COLUMN` の場合、依存 VIEW が `SELECT i.*` を使っていないか確認する

`last_synced_*_at` のような「変わっても `updated_at` を進めたくないメタデータカラム」を新設する場合だけ、`update_updated_at()` の `sync_metadata_keys` へ追加する。

## 5. 復旧確認の標準手順

同期系 migration の適用後は、SQL Editor で最低限これを確認する。

```sql
SELECT pg_get_functiondef(oid)
FROM pg_proc
WHERE proname = 'update_updated_at';
```

```sql
SELECT COUNT(*) AS recent_failures
FROM sync_failures
WHERE last_seen_at > NOW() - INTERVAL '15 minutes'
  AND resolved_at IS NULL;
```

```sql
SELECT COUNT(*) AS auto_transition_candidates
FROM illustrators
WHERE master_status = '連絡中'::master_status_enum
  AND contacted_at IS NOT NULL
  AND contacted_at <= (CURRENT_DATE - INTERVAL '14 days');
```

判定基準:

- `update_updated_at()` が `to_jsonb(NEW)` / `to_jsonb(OLD)` 方式になっている
- 直近15分に再発した未解決 `sync_failures` が 0
- auto-transition 対象件数が通常範囲（数件から数十件程度）

## 6. 残るリスク

この対策で、削除済みカラム参照によって UPDATE 全体が失敗する同種事故は構造的に防げる。

ただし以下は別リスクとして残る。

- Notion / Sheets 側のプロパティ名変更
- ENUM に存在しない値の追加
- GitHub Actions の secrets 不備
- Notion API / Google Sheets API の一時障害
- `sync_failures` の未解決行が放置される運用リスク

これらは `sync_failures` と GitHub Actions の実行結果で検知し、個別に対処する。
