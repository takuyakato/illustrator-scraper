## 1. 概要

### 1.1 目的

既存のNotion「イラストレーター情報DB」と「Berryfeel-イラストレーター」別DBをSupabaseに統合移行し、同時にデータをクリーンアップする。

- 既存データを**全件**Supabaseに取り込む（欠損なし）
- 3つ並存しているステータス系プロパティ（連絡状況 / ステータス 1 / ステータス）を**マスターステータス（7値）**に統合する
- Berryfeel別DBをメインDBに吸収する
- Notion側は運用ビューで使うプロパティのみ残し、旧プロパティは非表示化する
- マイグレーション後は Supabase ↔ Notion の 10分おき同期を起動する

| Berryfeel-イラストレーター（別DB） | `2cac5063-7a7b-808e-889c-fef08afa0f73` | メインに吸収してアーカイブ |

## 4. マスターステータス変換マップ

### 4.1 変換ロジックの優先順位

複数のステータス系プロパティに値が入っている場合、以下の優先順位で評価する。

```
優先順位: 連絡状況 > ステータス 1 > ステータス > デフォルト
```

- **連絡状況**（status型）に値があれば、それを優先採用
- **連絡状況**が「未連絡」もしくは空の場合、**ステータス 1** を参照
- **ステータス 1** も空の場合、**ステータス**（select型、旧ネーム/線画用）を参照
- 全て空の場合は **候補** を設定

### 4.2 完全変換表

| 元プロパティ | 元の値 | → 新マスターステータス | 備考 |
|---|---|---|---|
| 連絡状況 | 連絡しない | 依頼不可 | |
| 連絡状況 | 未連絡 | **（下位参照）** | ステータス 1 を見にいく |
| 連絡状況 | 連絡中 | 連絡中 | |
| 連絡状況 | 時間を空けて再連絡 | 多忙辞退 | |
| 連絡状況 | 依頼失敗 | 依頼不可 | |
| 連絡状況 | 依頼成功 | 依頼成功 | |
| ステータス 1 | 完了 | 依頼成功 | |
| ステータス 1 | 依頼中 | 連絡中 | |
| ステータス 1 | 返信あり・依頼できそう | 連絡中 | |
| ステータス 1 | 未返信 | 返信なし | |
| ステータス 1 | 連絡したい | 候補 | |
| ステータス 1 | スケジュール確保済み | 連絡中 | |
| ステータス 1 | 依頼前 | 連絡中 | |
| ステータス 1 | 再連絡 | 多忙辞退 | 時間を置いて再アプローチ想定 |
| ステータス 1 | 依頼不可 | 依頼不可 | |
| ステータス 1 | 多忙のため辞退 | 多忙辞退 | |
| ステータス | ネーム:依頼中 | 連絡中 | |
| ステータス | 線画:依頼中 | 連絡中 | |
| ステータス | ネーム:頼まない | 依頼不可 | |
| ステータス | 線画:頼まない | 依頼不可 | |
| ステータス | 未依頼 | 候補 | |
| ステータス | ネーム:継続希望 | 依頼成功 | |
| ステータス | 線画:継続希望 | 依頼成功 | |
| （全て空） | ― | 候補 | デフォルト |

### 4.3 変換ロジック（擬似コード）

```typescript
function resolveMasterStatus(row: NotionRow): MasterStatus {
  // 1. 連絡状況を最優先
  const contact = row['連絡状況'];
  if (contact && contact !== '未連絡') {
    return mapContactStatus(contact);
  }

  // 2. ステータス 1 を次に参照
  const status1 = row['ステータス 1'];
  if (status1) {
    return mapStatus1(status1);
  }

  // 3. ステータス（旧ネーム/線画）を参照
  const oldStatus = row['ステータス'];
  if (oldStatus) {
    return mapOldStatus(oldStatus);
  }

  // 4. デフォルト
  return '候補';
}
```

## 5. ランク処理

### 5.1 マイグレーション時の処理

**全レコード**に対して以下を実施。

| フィールド | 初期値 |
|---|---|
| `rank` | 旧「ランク」プロパティの値をそのままコピー（S/A/B/C、空ならNULL） |
| `owner_confirmed_by` | `[]`（空配列、全員要確認扱い） |

旧ランク値はそのまま `rank` に入るだけで、専用の履歴カラムには残さない。変更履歴を追跡したい場合は **`migration_snapshot`（JSONB）** にマイグレーション時点の完全なスナップショットが保存されているので、そこから参照する。

### 5.2 運用開始後の変化

- オーナー（北條・三村・加藤）が「オーナー確認=空」のレコードを順次見直す
- ランクが正しければ `owner_confirmed_by` に自分の名前を追加
- ランクを変更したければ `rank` を変更 + `owner_confirmed_by` に自分の名前を追加

### 5.3 変更前のランクを参照したい場合

日常運用ではランクの変更履歴を別カラムに保持しない（シンプル運用優先）。必要な場合のみ Supabase Studio から `migration_snapshot` をクエリして参照する。

```sql
-- 例：変更前の旧ランクを参照
SELECT
  artist_name,
  rank AS 現在のランク,
  migration_snapshot->'notion_page'->'properties'->'ランク'->'select'->>'name' AS マイグレーション時の旧ランク
FROM illustrators
WHERE id = '<対象ID>';
```

## 6. Berryfeel別DB統合手順

### 6.1 処理方針

1. Berryfeel別DBの全レコードを取得
2. メインDBと**作家名 or メールアドレス**で突合
3. ケース別に処理
4. 別DBは削除せず、ページ名の先頭に `[ARCHIVED 2026-04-XX]` を付けて識別

### 6.2 ケース分け

| ケース | 状態 | 処理 |
|---|---|---|
| A | Berryfeel別DBにのみ存在 | Supabase.illustratorsに新規INSERT、`genres = ['Berryfeel']`、**`rank = 'S'`**、**`owner_confirmed_by = ['北條']`** |
| B | 両方に存在（同一人物） | メインDBのSupabaseレコードに `genres` に `'Berryfeel'` を追加、**`rank = 'S'` で上書き**、**`owner_confirmed_by` にも `'北條'` を追加**（重複排除） |
| C | 両方に存在（別人で同名） | 別レコードとして扱う（手動で判断）。ランクとオーナー確認はケースAと同じ扱い |

### 6.2.1 ランク=S、オーナー確認=北條 の根拠

BerryFeel別DBは北條さんが主に管理してきた経緯があり、**BerryFeelに採用された＝最高ランク相当（S）・画力確認済み** と見なせる。

- **ランク=S**: BerryFeel採用済みはプロジェクト対応の最上位（BerryFeelのみで依頼できる）に該当。既存メインDBに別のランクがあっても、BerryFeelに載っているという事実を優先して S で上書きする
- **オーナー確認=['北條']**: 北條さんが再レビューする必要がないことを示す。マイグレーション後すぐに「依頼可能プール」へ進める状態にするため。三村・加藤のレビューは必要になった時点で追加すればよい

### 6.3 突合ロジック（擬似コード）

```typescript
async function mergeBerryfeelDb() {
  const berryfeelRecords = await notion.queryAll(BERRYFEEL_DB_ID);
  const supabaseRecords = await supabase.from('illustrators').select('*');

  for (const bf of berryfeelRecords) {
    // 名前一致
    const byName = supabaseRecords.find(r => r.artist_name === bf.artistName);
    // メール一致
    const byEmail = bf.email ? supabaseRecords.find(r => r.email === bf.email) : null;

    const matched = byEmail || byName;

    if (matched) {
      // ケースB: ジャンル+Berryfeel、ランク=S で上書き、オーナー確認に '北條' を追加
      const genres = new Set(matched.genres ?? []);
      genres.add('Berryfeel');

      const owners = new Set(matched.owner_confirmed_by ?? []);
      owners.add('北條');  // 北條さん確認済み扱い（重複排除）

      await supabase.from('illustrators')
        .update({
          genres: Array.from(genres),
          rank: 'S',  // BerryFeel採用済み = 最高ランク相当、既存ランクがあってもSで上書き
          owner_confirmed_by: Array.from(owners),
        })
        .eq('id', matched.id);
    } else {
      // ケースA: 新規INSERT（ランク=S、オーナー確認=['北條'] 付き）
      await supabase.from('illustrators').insert({
        artist_name: bf.artistName,
        genres: ['Berryfeel'],
        rank: 'S',  // BerryFeel採用済み = 最高ランク相当
        owner_confirmed_by: ['北條'],  // 北條さん確認済み扱い
        migration_snapshot: { source: 'berryfeel_db', original: bf.raw },
        // ... その他フィールド
      });
    }
  }
}
```

```typescript
async function main() {

  console.log(`取得レコード数: ${allRecords.length}`);

  // 2. 各レコードを変換
  const transformed = allRecords.map((row) => {
    const xUrlRaw = row.properties['Xリンク']?.url ?? '';
    const xUsername = normalizeXUrl(xUrlRaw);

    const emailMain = row.properties['メール']?.email;
    const emailAlt = row.properties['メアド']?.email;

    //     かつビュー表示時に空行だと識別できなくなるため。
    //     「(名無し-XXXXXXXX)」の XXXXXXXX は Notion page ID の先頭8文字。
    const rawArtistName = row.properties['作家名']?.title?.[0]?.plain_text ?? '';
    const artistName = rawArtistName.trim() !== ''
      ? rawArtistName
      : `(名無し-${row.id.slice(0, 8)})`;

    return {
      notion_page_id: row.id,
      artist_name: artistName,

      // マスターステータス変換（Section 4）

      // ランク処理（Section 5）

      legacy_mail_alt: emailAlt ?? null,

      // クレジット名義変換（Section 8）※ convertCreditName は artistName も引数に取る

      // ジャンル
      genres: row.properties['ジャンル']?.multi_select?.map((s) => s.name) ?? [],

      // 非表示カラム（Supabaseのみ保持）
      legacy_status: row.properties['ステータス']?.select?.name ?? null,
      legacy_status_1: row.properties['ステータス 1']?.status?.name ?? null,
      legacy_contact_status: row.properties['連絡状況']?.status?.name ?? null,
      legacy_capuri_berryfeel_search:
        row.properties['Capuri/BerryFeel探し']?.multi_select?.map((s) => s.name) ?? [],
      legacy_mimura_comment: extractRichText(row.properties['三村コメント']),
      legacy_hojo_comment: extractRichText(row.properties['北條コメント']),
      legacy_mimura_points: row.properties['三村点数']?.number ?? null,
      legacy_hojo_points: row.properties['北條点数']?.number ?? null,
      legacy_found_date: row.properties['見つけた日']?.date?.start ?? null,
      legacy_found_by: row.properties['見つけた人']?.select?.name ?? null,
      legacy_start_date: row.properties['開始日']?.date?.start ?? null,
      legacy_end_date: row.properties['終了予定日']?.date?.start ?? null,

      // 再連絡時期は rich_text 型（自由記述）
      legacy_recontact_time:
        row.properties['再連絡時期']?.rich_text?.map((r) => r.plain_text).join('') ?? null,

      // 断られた理由は multi_select 型（配列で保存）
      legacy_rejection_reason:

      note: extractRichText(row.properties['備考']) ?? '',

      is_illustrator: true, // 既存レコードは全員イラストレーター扱い

      // 完全スナップショット（ロールバック用）
      migration_snapshot: { source: 'main_db', notion_page: row },
    };
  });

  // 3. バルクインサート（500件ずつ）
  const CHUNK = 500;
  for (let i = 0; i < transformed.length; i += CHUNK) {
    const chunk = transformed.slice(i, i + CHUNK);
    const { error } = await supabase.from('illustrators').insert(chunk);
    if (error) throw error;
    console.log(`インサート完了: ${i + chunk.length}/${transformed.length}`);
  }

  console.log('マイグレーション完了');
}

main().catch((e) => {
  console.error(e);
  process.exit(1);
});
```

### 7.3 Xリンク正規化関数（x-url-normalizer.ts）

```typescript
/**
 * Xリンクを正規化して x_username（小文字、@なし、スラッシュなし）を返す。
 */
export function normalizeXUrl(input: string | null | undefined): string | null {
  if (!input) return null;

  let s = input.trim();

  // @ から始まる場合
  if (s.startsWith('@')) return s.slice(1).toLowerCase();

  // URL 形式
  try {
    const url = new URL(s);
    // x.com / twitter.com / mobile.twitter.com
    if (!/^(x\.com|twitter\.com|mobile\.twitter\.com)$/.test(url.hostname)) {
      return null;
    }
    const username = url.pathname.split('/').filter(Boolean)[0];
    return username ? username.toLowerCase() : null;
  } catch {
    // URL でない単独文字列
    return s.replace(/^@/, '').toLowerCase();
  }
}
```

### 7.4 クレジット希望→クレジット名義の変換

既存「クレジット希望」（select）は「する / しない / 別名希望」等のカテゴリ値。

| 元の値 | → 新クレジット名義 |
|---|---|
| する（またはそれに類する値） | （作家名と同じ文字列） |
| しない | 空文字 |
| 別名希望 | 空文字（オーナーが後で手入力） |
| 空 | 空文字 |

```typescript
function convertCreditName(creditChoice: string, artistName: string): string {

  if (creditChoice === '別名希望') return ''; // 要手入力
```

### 7.5 ポートフォリオリンクとその他連絡先の振り分け

既存の「その他連絡先」カラムには、実際はポートフォリオサービス（lit.link / potofu.me 等）のURLが多数混在している。マイグレーション時に以下のパターンでURLを振り分け、`portfolio_link` と `other_contact` の両方を同時に初期化する。

| URLパターン | 振り分け先 | 例 |
|---|---|---|
| `lit.link/*` | `portfolio_link` | `https://lit.link/example` |
| `potofu.me/*` / `potofu.jp/*` | `portfolio_link` | `https://potofu.me/example` |
| `skeb.jp/*` | `portfolio_link` | `https://skeb.jp/@example` |
| `booth.pm/*` | `portfolio_link` | `https://example.booth.pm/` |
| 上記以外 | `other_contact`（元のまま） | 個人サイト・Misskey 等 |

```typescript
// scripts/migration/lib/portfolio-splitter.ts

/**
 * 「その他連絡先」のURLを portfolio_link / other_contact に振り分ける。
 */
export function splitPortfolioAndOther(
  url: string | null
): { portfolio: string | null; other: string | null } {
  if (!url) return { portfolio: null, other: null };

  const lower = url.toLowerCase();

  // ポートフォリオ系サービスのパターン
  if (
    lower.includes('lit.link') ||
    lower.includes('potofu.me') ||
    lower.includes('potofu.jp') ||
    lower.includes('skeb.jp') ||
    lower.includes('booth.pm')
  ) {
    return { portfolio: url, other: null };
  }

  // それ以外は other_contact として残す
  return { portfolio: null, other: url };
}
```

7.2 の擬似コード内では、`row.properties['その他連絡先']?.url` を `splitPortfolioAndOther()` に通し、戻り値を `portfolio_link` と `other_contact` のそれぞれに格納している。

## 11. マイグレーション後のオーナー初期作業

マイグレーション完了後、オーナー（北條・三村・加藤）は**ビュー1「オーナー確認用」**で以下の優先順位で確認を進める。

### 11.1 優先順位

マイグレーション直後は、`rank` に旧ランクの値がそのまま入っている。オーナーはまずランクを見て、シード候補を優先確認する。

| 優先度 | 条件 | 理由 |
|---|---|---|
| 🔥 最優先 | マスターステータス=依頼成功 かつ ランク=S | シード候補、スクレイパーの精度を左右する |
| ⭐ 高 | マスターステータス=依頼成功 かつ ランク=A | シード候補（BerryFeel/CAPURI向け） |
| 中 | マスターステータス=依頼成功 かつ ランク=B/C | 参考情報、B→Aへの格上げ候補 |
| 低 | マスターステータス=候補/多忙辞退/条件次第 かつ ランクあり | 再アプローチ検討時に参照 |
| 最低 | マスターステータス=依頼不可 | 原則触らない（ランクが明らかにおかしい場合のみ修正） |

### 11.2 オーナーの具体アクション

各レコードに対して以下のいずれか：

1. **ランクが正しい場合**
   - `owner_confirmed_by` に自分の名前を追加（マルチ選択で自分をON）
   - `rank` はそのまま

2. **ランクを変更したい場合**
   - `rank` を新しい値に変更
   - `owner_confirmed_by` に自分の名前を追加
   - （変更前の値を後で確認したくなった場合は、`migration_snapshot` から参照可能）

3. **絵柄タグが把握できる場合**
   - `絵柄タグ` にイケメン/リアル/デフォルメ/クセ強 を付与（最大複数）
   - 混在する場合は空のままでOK

### 11.3 進捗管理

週1の定例で進捗確認：

```sql
-- オーナー確認進捗
SELECT
  (SELECT COUNT(*) FROM illustrators WHERE array_length(owner_confirmed_by, 1) > 0) as 確認済み,
  (SELECT COUNT(*) FROM illustrators WHERE array_length(owner_confirmed_by, 1) IS NULL OR array_length(owner_confirmed_by, 1) = 0) as 未確認,
  (SELECT COUNT(*) FROM illustrators) as 合計;
```

目標：最優先（🔥）と高優先度（⭐）は**マイグレーション後2週間以内に全件確認**する。

## 12. 注意事項と想定トラブル

### 12.1 Notion API のレート制限

- Notion APIは 3 requests/sec の制限
- スクリプト内で 400ms 程度の sleep を挟む
- バルク処理（例: Step 10-1 の全ページ更新）は時間がかかる（数百件で数分）

### 12.2 Notion status 型プロパティの同期

- Notion の `status` 型はオプション名を事前に作成しておく必要がある
- Step 1 でマスターステータスの7値を全て追加してからStep 10-1 を実行する
- 存在しない status 値を APIで指定すると 400 エラー

### 12.3 作家名の重複

- 「作家名」は一意ではない（同名異人がいる可能性）
- マイグレーションでは `notion_page_id` を一意キーとして使う
- Berryfeel統合時はメール優先で突合する

### 12.4 空のXリンク

- Berryfeel別DBにはXリンクがないレコードが多い
- `x_username IS NULL` を許容する（UNIQUE制約は `WHERE x_username IS NOT NULL` を付けた部分インデックスにする）

### 12.5 他メンバーの同時編集

- マイグレーション中は北條・三村さんにDB編集を止めてもらう
- Step 2（エクスポート）〜Step 10完了までの時間帯を事前に伝える

### 12.6 「連絡した人」の型変更に伴う一時的な併存

- Step 1-B で新プロパティ「連絡した人_new」を作成するが、旧「連絡した人」（select）は**削除せず残す**
- 運用開始後しばらくは両方存在するが、ビューで旧を非表示にすることで混乱を防ぐ
- 最終的な旧プロパティ削除は、運用1ヶ月後などに改めて判断
