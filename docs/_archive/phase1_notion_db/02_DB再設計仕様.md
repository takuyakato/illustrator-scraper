# Phase1-02: Notion DB 再設計仕様

**バージョン**: 1.0
**作成日**: 2026-04-21
**対象DB**: [イラストレーター情報](https://www.notion.so/e72fd069f4284a4a946ed8c61f115345)

---

## このドキュメントの役割

「01_現状分析とギャップ」で特定したギャップを埋めるための**DB再設計仕様**を定義する。マイグレーション手順は「03_マイグレーション手順」を参照。

---

## 1. 設計方針

1. **単一ライフサイクル原則**: 作家の状態は `マスターステータス` 1つで表現する
2. **スクレイパー友好**: スクレイパーが読む/書くプロパティは命名を揃える
3. **非破壊的整理**: 既存の有用データは失わない（重複だけ解消）
4. **運用マニュアルと整合**: 運用マニュアルv1.0が前提とする構造に寄せる

---

## 2. 再設計後のプロパティ一覧

### 2-1. コア情報（必須）

| プロパティ名 | 型 | 入力者 | 説明 |
|---|---|---|---|
| 作家名 | title | スカウト/オーナー | 作家の識別名（通称可） |
| ID | unique_id (NO-xxx) | 自動 | 自動採番 |
| Xリンク | url | スカウト（GAS経由） | `https://x.com/username` |
| Pixivリンク | url | スカウト（GAS経由） | |
| その他連絡先 | url | スカウト/オーナー | lit.link・Potofu等 |
| メール | email | 連絡担当 | 連絡用メールアドレス |

### 2-2. ステータス（単一化）

| プロパティ名 | 型 | 値 | 入力者 |
|---|---|---|---|
| **マスターステータス**（新設） | status | 下記参照 | GAS/オーナー/連絡担当 |

**マスターステータスの値**（ライフサイクル順）:

```
┌─ To-do ────────────────┐
│  候補                   │ ← GASが転記時にセット
│  依頼可能               │ ← オーナーが画力確認後
│  要連絡                 │ ← 連絡担当がピックアップ
└────────────────────────┘
┌─ In progress ──────────┐
│  連絡中                 │
│  返信あり               │
│  依頼中                 │
└────────────────────────┘
┌─ Complete ─────────────┐
│  依頼成功               │ ← シード抽出対象
│  殿堂入り               │ ← （将来用・当面使わない）
│  多忙辞退               │
│  連絡不通               │
│  依頼不可               │
│  連絡しない             │ ← 自主的に対象外としたケース
└────────────────────────┘
```

### 2-3. 画力・適性

| プロパティ名 | 型 | 値 | 入力者 |
|---|---|---|---|
| ランク | select | S / A / B / C | オーナー（確定時） |
| **TL適性**（新設） | select | ◎ / ○ / △ / × | オーナー |
| ジャンル | multi_select | BLサンド / Capuri / Webtoon / Berryfeel / アシスタント | オーナー |

⚠️ プロパティ名は実DBの既存「ランク」を踏襲（リネームしない）。スクレイパー側で参照するときもこの名前を使う。

## 4. 旧ステータス → マスターステータス 変換マップ

マイグレーション時に、既存の3プロパティから新マスターステータスへ変換する規則。

### 優先順位

`連絡状況` > `ステータス 1` > `ステータス` の順で優先し、最上位の値を採用する。

### 変換表

| 元プロパティ | 元の値 | → マスターステータス |
|---|---|---|
| 連絡状況 | 依頼成功 | **依頼成功** |
| 連絡状況 | 連絡中 | **連絡中** |
| 連絡状況 | 時間を空けて再連絡 | **要連絡** |
| 連絡状況 | 依頼失敗 | **連絡不通** |
| 連絡状況 | 連絡しない | **連絡しない** |
| 連絡状況 | 未連絡 | （下位へ委譲） |
| ステータス 1 | 完了 | **依頼成功** |
| ステータス 1 | 依頼中 | **依頼中** |
| ステータス 1 | 返信あり/依頼できそう | **返信あり** |
| ステータス 1 | 未返信 | **連絡中** |
| ステータス 1 | 連絡したい | **要連絡** |
| ステータス 1 | スケジュール確保済み | **依頼可能** |
| ステータス 1 | 依頼前 | **依頼可能** |
| ステータス 1 | 再連絡 | **要連絡** |
| ステータス 1 | 依頼不可 | **依頼不可** |
| ステータス 1 | 多忙のため辞退 | **多忙辞退** |
| ステータス | ネーム:依頼中 / 線画:依頼中 | **依頼中** |
| ステータス | ネーム:頼まない / 線画:頼まない | **依頼不可** |
| ステータス | 未依頼 | **依頼可能** |
| ステータス | ネーム:継続希望 / 線画:継続希望 | **依頼成功** |
| （すべて未設定） | | **候補** |

⚠️ この変換は**機械的マッピング**に過ぎない。マイグレーション後、オーナーが上位50件をざっと見直すステップを推奨。

## 8. ビュー推奨設定

DB再設計後、以下のビューを用意しておくと運用が楽になる。

| ビュー名 | フィルタ | 用途 |
|---|---|---|
| 🆕 候補（要判定） | マスターステータス=候補 | オーナーが見るプール |
| ⭐ 依頼可能プール | マスターステータス=依頼可能 | 連絡担当が選ぶ元 |
| 🌱 シード候補 | ランク IN (S, A) かつ マスターステータス=依頼成功 | 週次バッチが読む範囲 |
| 🚩 要連絡 | マスターステータス=要連絡 | 連絡担当のTODO |
| 💤 連絡不通・辞退 | マスターステータス IN (連絡不通, 多忙辞退, 依頼不可) | アーカイブ的確認用 |
| ⚠️ 未設定あり | マスターステータス=空 OR ランク=空 | 整理漏れチェック |
