## 4. 主要ギャップ

### 4-1. プロパティ名の不一致

| スクレイパー側が期待 | 実DBの該当 | 対応 |
|---|---|---|
| Xアカウント | `Xリンク` | 実DB側を正とする（スクレイパー側をリネーム） |
| 画力ランク | `ランク` | **要判断**: どちらに寄せるか |
| マスターステータス | 存在しない | **新設** |
| TL適性 | 存在しない | **新設** |
| 検出元アカウント | 存在しない | **新設** |

### 4-2. ステータス体系の混乱（最重要課題）

現状は `ステータス` / `ステータス 1` / `連絡状況` の3つが並存しており、スクレイパーが参照すべき「作家のライフサイクル」が単一プロパティで取得できない。

**シード抽出条件「ランクS/A かつ 連絡状況=依頼成功」** は、実DBの `連絡状況=依頼成功` で部分的に実現可能だが、以下のリスクあり：

- `連絡状況` がどの程度運用されているか不明（`未連絡` のまま放置されているレコードが多い可能性）
- 新規追加レコードで「候補」段階の扱いが定義されていない
- `ステータス 1` と `連絡状況` のどちらを正とするか決まっていない

### 4-3. メール重複（`メール` と `メアド`）

どちらを主にして使うか未定。片方に統合する。

### 4-4. Berryfeel別DBの存在

メインDBと独立しており、**作家の一元管理ができていない**。スクレイパーが重複チェックを行っても、別DBのレコードは検出できない。

### 4-5. `三村点数`・`北條点数` の運用ルール不在

数値プロパティとして存在するが、運用マニュアルに記載なし。何のための点数か、誰がいつ入れるかが不明確。

### 4-6. `ジャンル` と `Capuri/BerryFeel探し` の重複

ジャンルに `Capuri` `Berryfeel` が含まれており、専用の `Capuri/BerryFeel探し` プロパティと役割が重複。

## 6. ギャップを埋めるアプローチ（次ドキュメントで詳細化）

Phase1で行う作業の方向性：

1. **プロパティ追加**
   - マスターステータス（新設・status型）
   - TL適性（新設・select型）
   - 検出元アカウント（新設・rich_text型）

2. **プロパティ整理**
   - `ステータス` + `ステータス 1` + `連絡状況` → **マスターステータス** に一本化
   - `メール` + `メアド` → `メール` に統合
   - `ジャンル` と `Capuri/BerryFeel探し` → `ジャンル` に一本化

3. **別DB統合**
   - `Berryfeel-イラストレーター` のレコードをメインDBに移行
   - ジャンル=Berryfeel タグで識別

4. **既存レコードのステータス変換**
   - 既存の3ステータスから、新マスターステータスへのマッピングルールを定義

5. **インテグレーション接続**
   - `Claude.Bisque` をメインDBに接続
